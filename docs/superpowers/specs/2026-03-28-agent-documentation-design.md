# Agent Documentation Design

**项目：** AI Video Factory Pro  
**日期：** 2026-03-28  
**状态：** Draft reviewed with user

---

## 1. 背景

当前项目已经有一组运行时 agent，分别负责：

- 主编排
- 剧本解析
- 角色设定
- Prompt 生成
- 一致性检查
- 配音
- 视频合成

这些能力目前主要通过源码和 README 被理解。这样做的问题是：

- 新人或 AI 代理需要读多个文件才能建立全局认知
- 复杂 agent 的职责边界不够显式
- 规则、流程、输入输出关系分散在实现代码里
- 一些 agent 值得单独解释，但“一 agent 一文档”又会造成文档膨胀

因此，需要建立一套专门面向“快速理解项目 + 方便维护”的 agent 文档层。

---

## 2. 目标

本设计的目标是：

- 让开发者或 AI 代理可以快速理解当前项目里的 agent 体系
- 为复杂 agent 提供稳定、可维护的说明文档
- 降低必须从源码逆向理解主流程的成本
- 在文档数量和可维护性之间取得平衡

---

## 3. 非目标

本设计**不**做以下事情：

- 不为每个 agent 机械地单独写一份文档
- 不替代 `skills/project/` 的规则层文档
- 不迁移运行时代码
- 不做跨项目通用 agent 文档模板设计

本设计只服务于**当前项目的 agent 理解与维护**。

---

## 4. 设计结论

推荐采用：

**1 份总览 + 3 份核心链路文档 + 1 份合成文档**

推荐结构如下：

```text
docs/
  agents/
    README.md
    director.md
    script-parser.md
    visual-design.md
    video-composer.md
```

这比“一 agent 一文档”更合适，原因是：

- 总览文档负责快速建立全局地图
- 核心复杂链路有单独文档，方便深入理解
- 简单执行层 agent 不必为了文档而文档
- 文档数量控制在合理范围内，维护成本低

---

## 5. 为什么不做“一 agent 一文档”

虽然“一 agent 一文档”看起来整齐，但对当前项目不是最佳方案。

### 问题 1：小 agent 文档容易很空

例如执行层较轻的 agent，如果单独拆出文档，常常只会写出：

- 它调用了哪个 API
- 它输出什么文件

这种文档信息密度不高，后续很容易过期。

### 问题 2：链路型知识会被拆散

例如：

- 角色设定
- Prompt 生成
- 一致性检查

这三者在项目里天然属于一条视觉设计链路。如果强拆成三份独立文档，理解成本反而会上升。

### 问题 3：维护成本高

每增加一份文档，就多一份漂移风险。

因此，当前项目更适合：

- 给复杂、关键、边界清晰的部分写独立文档
- 给轻量、执行型部分保留总览级说明

---

## 6. 推荐文档结构

### 6.1 `docs/agents/README.md`

**定位：** agent 总览入口文档。

**负责内容：**

- 当前系统里有哪些 agent
- 每个 agent 一句话职责
- 主流程调用顺序
- 输入输出关系
- 哪些 agent 已经有 project skill 支撑

**价值：**

- 最快建立项目全局认知
- 降低新人和 AI 代理理解门槛
- 提供跳转入口到更细的 agent 文档

---

### 6.2 `docs/agents/director.md`

**定位：** 主编排文档。

**负责内容：**

- `director` 的职责边界
- 主流程步骤说明
- 状态文件与断点续跑逻辑
- 错误处理和缓存策略
- `director` 不负责什么

**为什么值得单独写：**

- 它是整个系统的编排入口
- 强依赖任务状态、恢复、异常和输出路径管理
- 是最容易成为维护中心的文件之一

---

### 6.3 `docs/agents/script-parser.md`

**定位：** 上游数据结构说明文档。

**负责内容：**

- 剧本输入预期
- 分镜输出 JSON 结构
- 字段约束和默认值规则
- 常见解析异常
- 下游 agent 如何依赖这些字段

**为什么值得单独写：**

- 它定义了整条流水线最上游的数据契约
- 后续很多问题都会回溯到这里

---

### 6.4 `docs/agents/visual-design.md`

**定位：** 视觉设计链路文档。

**覆盖范围：**

- `characterRegistry`
- `promptEngineer`
- `consistencyChecker`

**负责内容：**

- 角色视觉档案如何生成
- Prompt 如何拼接
- 风格词、镜头词、质量词如何组织
- 一致性检查结果如何转成修正建议
- 三者之间的输入输出关系

**为什么合并写：**

- 这三者天然是一条视觉设计链路
- 合并写比拆成 3 份更利于理解上下游关系
- 可以减少重复描述

---

### 6.5 `docs/agents/video-composer.md`

**定位：** 最终渲染与合成文档。

**负责内容：**

- composition plan
- 音频时间线规划
- 字幕时间轴规则
- FFmpeg 依赖与排障
- 平台兼容性注意事项

**为什么值得单独写：**

- 这部分复杂度高且容易出错
- 合成问题通常需要独立排查
- 与前面的内容型 agent 边界明显

---

## 7. 不建议单独写文档的 agent

当前阶段不建议单独给以下 agent 各写一份独立 md：

- `imageGenerator`
- `ttsAgent`
- 单独的 `consistencyChecker`

原因如下：

- 更偏执行层，规则密度还不够高
- 内容容易和现有 `skills/project/` 重复
- 单独成文的边际收益不高

这些部分可以先通过：

- `docs/agents/README.md` 的总览说明
- `visual-design.md` 的链路说明
- 对应 project skill

来覆盖。

---

## 8. 每份文档建议结构

除总览外，单个 agent 文档建议统一包含：

```md
# Agent Name

## 职责

## 输入

## 输出

## 主流程 / 处理步骤

## 依赖关系

## 常见问题

## 不负责的内容

## 来源文件

## 相关 skill
```

### 结构原则

- 优先写边界，而不是代码细节
- 优先写输入输出契约，而不是实现顺序复述
- 对复杂链路要写清楚上下游依赖
- 标明来源文件，避免文档漂移

---

## 9. 与 `skills/project/` 的关系

`docs/agents/` 和 `skills/project/` 并不冲突，两者职责不同。

### `docs/agents/`

回答的是：

- 这个 agent 在项目里做什么
- 和其它 agent 的关系是什么
- 输入输出怎么流转
- 维护时要注意什么

### `skills/project/`

回答的是：

- 某一类项目知识的规则是什么
- 什么时候使用这套规则
- 应该遵守哪些约束

简单说：

- `docs/agents/` = 理解项目结构
- `skills/project/` = 复用项目规则

---

## 10. 落地顺序

推荐按以下顺序落地：

### 阶段 1：`docs/agents/README.md`

优先级最高，因为它最直接提升全局理解速度。

### 阶段 2：`docs/agents/director.md`

第二优先级，因为所有人最后都会回到这里看主流程。

### 阶段 3：`docs/agents/script-parser.md`

第三优先级，因为它定义上游数据契约。

### 阶段 4：`docs/agents/visual-design.md`

第四优先级，用链路方式整理视觉相关三个 agent。

### 阶段 5：`docs/agents/video-composer.md`

第五优先级，因为当前合成逻辑仍在迭代，晚一点写更稳。

---

## 11. 风险与维护策略

### 风险 1：文档与代码漂移

**问题：**
文档写完后，代码改了但文档没更新。

**对策：**

- 每份文档写清楚 `来源文件`
- 后续改动 agent 边界或字段时，同步更新对应文档
- 把“是否需要更新 docs/agents”纳入变更 checklist

### 风险 2：文档和 skill 重复

**问题：**
同样内容在 `docs/agents` 和 `skills/project` 里重复维护。

**对策：**

- `docs/agents` 侧重职责、流程、关系
- `skills/project` 侧重规则、模板、约束
- 两边互相引用，但不要抄写彼此主体内容

### 风险 3：文档拆分过细

**问题：**
文档越来越多，理解反而更困难。

**对策：**

- 坚持“总览 + 核心链路文档”模式
- 不机械执行“一 agent 一文档”
- 只有当某个 agent 复杂度明显上升时再单独拆出文档

---

## 12. 最终结论

当前项目**有必要**写 agent 文档，但**没有必要给每个 agent 都单独写一份 md**。

推荐方案是：

- 建立 `docs/agents/README.md` 作为总览
- 单独维护 4 份核心文档：
  - `director.md`
  - `script-parser.md`
  - `visual-design.md`
  - `video-composer.md`
- 轻量执行层 agent 暂不独立成文
- `docs/agents/` 与 `skills/project/` 并行存在，分别承担“结构理解”和“规则复用”职责

这是当前项目在理解效率、维护成本和长期稳定性之间最平衡的方案。
